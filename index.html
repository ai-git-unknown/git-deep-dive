1The big picture: the three states

Before any internals, internalise the model Git was built around. A file lives in one of three places at any moment:

Working Directory

Your actual files on disk — what you edit in an editor. Plain files.

Staging Area (Index)

A binary snapshot file (.git/index) of what the next commit will contain.

Repository (.git/objects)

The permanent, content-addressed database of every committed version, ever.

The fundamental loop is: you modify files in the working directory, you stage a snapshot into the index with git add, and you commit the index into the object database with git commit. That's the whole game. Everything else (branches, merges, remotes) is machinery built on top of that database.

# The core lifecycle
vim file.txt          # 1. edit  -> working directory
git add file.txt      # 2. stage -> index
git commit -m "msg"   # 3. store -> objects
Key insight: Git does not store diffs by default and it does not track files by name at the storage layer. It stores complete snapshots as immutable content-addressed objects. The "name" of a file is computed, not assigned.

2Inside .git: the anatomy

Run git init and Git creates one directory: .git. Almost everything Git knows lives here. The working tree is outside it.

.git/
├── HEAD            # pointer to current branch (e.g. ref: refs/heads/main)
├── config          # repo-local config
├── index           # the staging area (binary)
├── objects/        # the object database (blobs, trees, commits, tags)
│   ├── info/
│   └── pack/       # compressed packfiles
├── refs/           # references (branches, tags)
│   ├── heads/      # local branches: refs/heads/main
│   ├── tags/       # tag refs
│   └── remotes/    # remote-tracking branches
├── logs/           # reflogs (movement history of refs)
├── hooks/          # client/server scripts
└── packed-refs    # packed reference storage for performance

Notice what's not here: there is no "branch list" file, no "commit log" file. Branches are just tiny files in refs/heads/ containing a 40-char SHA-1 (or SHA-256) hash. The "log" is reconstructed by walking the commit graph from those hashes.

SHA-1 vs SHA-256: Traditional Git hashes objects with SHA-1 (160-bit). The sha256 hash algorithm is supported via extensions.objectformat in config, but the ecosystem still defaults to SHA-1. A hash collision is astronomically unlikely and Git has never had a real-world object collision in practice.

3The object database: content-addressable storage

This is the heart of Git. Git is a content-addressable filesystem: you give it content, it returns a key (the hash); later you give it the key, it returns the exact content. Four object types exist:

TypeWhat it storesIdentified by
blobRaw file contents (no name, no path)hash of "blob <size>\0<content>"
treeDirectory listing: names → hashes + modeshash of tree content
commitPointer to a tree, parent(s), author, messagehash of commit content
tagAnnotated tag object (points to a commit)hash of tag content

The object ID (OID) is computed as:

SHA1( header + " " + length + "\0" + content )

The header is the object type, a space, the byte-length of the content (not including header), a NUL byte, then the raw content. Stored objects are zlib-compressed, so on disk each object is zlib( header + \0 + content ). Loose objects live at objects/ab/cdef... — the first 2 hex chars are a directory shard, the rest is the filename. This two-level split avoids huge directories (filesystem performance).

Try it yourself (plumbing): git hash-object -w file.txt writes a blob and prints its hash — no commit, no staging, just raw storage. git cat-file -p <hash> prints the content back. This proves blobs have no filename.

4Blobs: file content, anonymous

A blob ("binary large object") holds the exact bytes of a file. Critically:

  • It stores no filename, no path, no permissions beyond a mode bit.
  • Identical content → identical hash → stored once, no matter how many files/commits reference it. This is deduplication for free.
  • An empty file and a file with different content produce different blobs.
# Two different paths, same content => ONE blob
echo "hello" > a.txt
echo "hello" > b.txt
git hash-object -w a.txt   # e.g. ce013625... 
git hash-object -w b.txt   # SAME hash — one object reused

One blob, many references. Git never stores the bytes twice.

5Trees: directories as snapshots

A tree object represents one directory. It is an ordered list of entries, each with: a file mode, the object type, the SHA of the child object, and the name. Trees reference blobs (for files) and other trees (for subdirectories). This is a Merkle tree — the tree's hash depends on the hashes of everything beneath it, recursively.

# git cat-file -p <tree-hash> looks like:
100644 blob ce013625...  a.txt
100644 blob ce013625...  b.txt
040000 tree 9d1c...      src/
100755 blob a1b2c3...    run.sh

A commit points to the root tree; the root tree points to blobs and sub-trees. The whole project state is one hash.

Why this matters: Changing one character in one file changes its blob hash → changes the containing tree → changes every parent tree up to root → changes the commit hash. A commit hash therefore uniquely and verifiably identifies the entire project state. Tamper with any historical byte and every downstream hash breaks — that's Git's integrity guarantee.

6Commits: the nodes of the graph

A commit object ties a tree to history. Its content:

tree <root-tree-hash>
parent <parent-commit-hash>     # omitted for the first commit; multiple for merges
author Name <email> <timestamp> <tz>
committer Name <email> <timestamp> <tz>

Commit message here

That's it. A commit is a pointer to a tree, plus zero or more parent pointers, plus metadata. History is a directed acyclic graph (DAG) where each commit knows its parent(s). There is no "list of commits" — there's a graph you walk backward from the tip.

Each arrow points parent-ward. git log is just a DFS walk of this DAG.

author vs committer: author is who wrote the change; committer is who applied it to the repo (they differ after a rebase, cherry-pick, or patch apply).

7The index / staging area: a third state

The index (.git/index) is a binary file that is a flat list of every file in the next commit, each entry storing: path, mode, the blob SHA, and stat data (mtime, size) for fast "is this file dirty?" checks without re-hashing.

Why it exists: it lets you build a commit incrementally and independently of your working directory. You can stage file A, keep editing file B, stage B later, and commit exactly the staged set — even if B on disk differs from what you staged.

git add file.txt     # computes blob, writes to objects/, records entry in index
git status           # compares: working dir vs index vs HEAD commit
Three-way mental model for git status: It shows the difference between (a) HEAD commit, (b) index, and (c) working tree. "Changes to be committed" = index ≠ HEAD. "Changes not staged" = working dir ≠ index.

git add does not "add a file to a changeset" in the CVS sense — it snapshots the current content of that file into the index. Running git add again overwrites the index entry with the newer content. The index is reusable and mutable until git commit freezes it into a tree.

8References & HEAD: where the graph is entered

Commit hashes are unwieldy, so Git gives them human names via references (refs). A ref is just a file containing a 40-char hash.

RefPathMeaning
Branchrefs/heads/mainPoints to the tip commit of a line of work. Moves on every commit.
Tag (light)refs/tags/v1.0Fixed pointer to a commit. Does not move.
Remote-trackingrefs/remotes/origin/mainCached view of a remote branch.
HEAD.git/HEADPoints to which ref is checked out (or directly to a commit = "detached HEAD").
# HEAD usually looks like this:
$ cat .git/HEAD
ref: refs/heads/main

# and the branch file holds the tip commit:
$ cat .git/refs/heads/main
3f2a9c1e... 

# So HEAD → main → commit. That's the "you are here" pin.
Detached HEAD: git checkout <commit-hash> makes HEAD point straight at a commit instead of a branch. New commits there aren't on any branch and can be lost when you switch away — unless you create a branch from them. The reflog still remembers them.

9Branching: cheap pointers, not copies

A branch is not a forked copy of files. It is a 41-byte file holding one commit hash. Creating a branch is O(1) — just write a new ref. This is why Git branches are free and ubiquitous.

git branch feature     # creates refs/heads/feature = same hash as current HEAD
git checkout feature    # moves HEAD to point at refs/heads/feature
# or in one step:
git checkout -b feature

Both refs point at the same commit. No file duplication. A new commit on feature just rewrites refs/heads/feature to the new hash; main stays put.

10Merging: fast-forward vs three-way

Fast-forward

If the target branch hasn't diverged (main is an ancestor of feature), merge just moves the pointer forward — no new commit, no DAG join.

Three-way merge

When both branches have new commits, Git finds the merge base — the nearest common ancestor of the two tips — and performs a three-way merge using three inputs: base (B), ours (O), theirs (T). For each file:

B → O → TResult
same → change → sametake the change (O)
same → same → changetake the change (T)
same → change → change (identical)take either
same → change → change (different)conflict — human resolves
base only on one sideadd/delete handled per case

The result is recorded as a merge commit with two parents (ours first, theirs second), permanently joining the histories.

Three inputs (root/base, main, feature) → merge commit with two parents. The merge base is the ancestor Git computes the diffs against.

Conflicts are not errors. A conflict means Git couldn't auto-decide. It writes both versions into the file with <<<<<<< / ======= / >>>>>>> markers, stages nothing for that path, and waits. Resolve, git add, then git commit.

11Rebase: replaying commits onto a new base

git rebase main does not move commits — it creates brand-new commits with new hashes, each a cherry-pick of your changes applied on top of the new base. Your old commits become orphaned (and eventually garbage-collected).

          A---B  feature          A'--B'  feature (after rebase)
         /              ==>      /
    C---D---E  main         C---D---E  main
  1. Find commits on feature not in main (A, B).
  2. Detach HEAD at main's tip (E).
  3. Sequentially cherry-pick A then B, producing A′ and B′ with fresh hashes.
  4. Repoint feature to B′.
Golden rule: Never rebase commits that have been pushed and shared with others. Rewriting history changes hashes; collaborators' histories will diverge and require forced, painful reconciliation. Rebase is for cleaning local history before sharing.

Interactive rebase (git rebase -i) extends this: it lets you pick, reword, edit, squash, or drop each commit in the todo list — all implemented as controlled replay.

12Remotes: sharing the object database

A remote is just another Git repository. git push and git fetch are neighbours in the object database — they ship objects over the wire and update refs.

  • fetch: downloads objects + updates refs/remotes/origin/* (remote-tracking branches). Never touches your working tree or local branches.
  • push: uploads objects missing on the remote, then asks the remote to move its ref to your commit. Rejected if non-fast-forward (would lose others' work) unless forced.
  • pull = fetch + merge (or rebase with --rebase).
The object protocol: Git computes the set of objects the other side lacks (via "want"/"have" negotiation), then packs and transfers only those. After transfer, both repos hold identical objects addressed by identical hashes — that's how two machines stay in sync without ever comparing file names.

13Packfiles & garbage collection

Storing every version as a separate zlib file would be huge. Git periodically packs loose objects into a packfile (objects/pack/*.pack) using delta compression: similar objects are stored as a base + a binary diff (delta). A companion .idx index maps hashes → offsets for fast lookup.

git gc                # repack loose objects, prune unreachable ones, pack refs

Garbage collection removes objects nothing points to (via refs, reflog, or the index). That's why deleted branches' commits linger until the reflog expires (default 90 days for reachable, 30 days for unreachable) and gc prunes them. This is your safety net against git reset --hard disasters.

Why git gc is usually automatic: Git triggers background GC when loose object count crosses a threshold (default 6700). You rarely run it by hand, but knowing it exists explains sudden disk savings and the disappearance of "lost" commits.

14Reflog: your undo button

The reflog records every move of a ref (HEAD and each branch) — every checkout, commit, reset, merge. It lives in .git/logs/ and is local only (never pushed).

git reflog            # shows HEAD's movement history with SHAs
git reflog show main  # branch-specific history

# Recover a "lost" commit after a bad reset:
git reset --hard HEAD@{1}     # go back one reflog entry
# or create a branch on the orphaned commit:
git branch recovered <sha>
This is why commits are rarely truly gone. Until GC prunes them (weeks later), any commit reachable from the reflog can be rescued. git fsck --lost-found finds dangling commits/objects the reflog might miss.

15Plumbing cheat sheet

The "porcelain" commands (add, commit, status) are user-friendly wrappers. The "plumbing" commands expose the raw machinery — perfect for truly understanding Git.

CommandWhat it reveals / does
git hash-object -w <file>Store raw content as a blob; print its hash.
git cat-file -p <sha>Print any object's content (blob/tree/commit/tag).
git cat-file -t <sha>Print an object's type.
git cat-file -s <sha>Print an object's size.
git write-treeTurn the current index into a tree object; print its hash.
git commit-tree <tree> -p <parent> -m "msg"Create a commit object manually.
git update-ref refs/heads/x <sha>Low-level branch creation/movement.
git rev-parse HEADResolve a ref/expression to a full SHA.
git ls-tree <tree>List a tree's entries (like ls for a snapshot).
git fsck --lost-foundFind dangling/unreachable objects.
Build a commit by hand once: hash-object a file → update-index --add → write-tree → commit-tree with a parent → update-ref HEAD's branch. Doing this manually cements the entire model in a way no tutorial can.

You now understand Git from the bytes up: a content-addressed object store (blobs, trees, commits, tags), a mutable index that stages the next tree, refs that name tips, a DAG you walk for history, and packfiles/reflog that keep it efficient and recoverable. Everything else is syntax.
